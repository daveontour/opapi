Running the Service

The service can be either run as:

·       Command prompt

·       Command prompt in performance test mode

·       Command prompt in demo mode

·       Windows Service

It is recommended to run it from the command prompt during initial deployment allow debugging of issues easier.

The functionality of the API is exactly the same in whatever mode the service is run

 

Running from the Command Prompt

In the directory where the system is installed, type

 

C:\>flightresourcerestapi.exe debug  (see below)

 

Running from the Command Prompt in Performance Test Mode

When run in performance test, the system does not need to connect to an instance of AMS.

In this mode, the system will use the configuration in test.json  to create an airport with checkin, gate, stand, carousel and  chute
resources in its  internal cache.

 It will also create the number of flights specified in the command line and allocate those flights to the configured resources.

In Performance Test Mode, Rabbit MQ is required to process test messages

This allows:

·       Demonstration of the system and API usage without a connection to AMS

·       Testing of the memory usage of the system for variable number of flights

·       Testing of API response times for variable number of flights

 

C:\>flightresourcerestapi.exe perftest 2000  (see below)

In this case 2000 is the number of flights we want to create, but this could be any number.

 The system creates arrival/departure pairs

Running from the Command Prompt in Demo Mode

When run in demo mode, the system does not need to connect to an instance of AMS.

In this mode, the system will use the configuration in test.json  to create an airport with checkin, gate, stand, carousel and  chute
resources in its  internal cache.

 It will also create the number of flights specified in the command line and allocate those flights to the configured resources.

The difference between Demo mode and Performance Test Mode is that in Demo Mode the system does NOT need to
to connect to a Rabbit MQ instance. Rabbit MQ is NOT needed in Demo Mode, it is a complete standalone mode of
operation that allows the API to be demonstrated and tested

This allows:

·       Demonstration of the system and API usage without a connection to AMS

·       Testing of the memory usage of the system for variable number of flights (this may be underestimated because Rabbit MQ is not used)

·       Testing of API response times for variable number of flights (This is not impacted by the absence of RabbitMQ)

 

C:\>flightresourcerestapi.exe perftest 2000  (see below)

In this case 2000 is the number of flights we want to create, but this could be any number.

 The system creates arrival/departure pairs

 

 

Running as a Windows Service

 

Configuration

Configuration of the service is controlled by three files:

·       airports.json

·       users.json

·       service.json

Each file is a JSON formatted file that is read by the service.

 

Airport Configuration

The airports.json file defines the AMS airports connected to the service. Multiple airports can be defined in the configuration.
The multiple airports can come from a multi airport instance configuration of AMS, or from separate instances of AMS.

The data from the different airports is not mixed or consolidated. The data from the different airports has to be accessed independently
and user access can be restricted to a subset of airports.

Example airports.json configuration file

{

    "airports":[

        {

            "airport":"ABC",

            "token":"0ab7d73d-e93a-480b-ba8c-a35943161cb0",

            "url":"http://localhost/SITAAMSIntegrationService/v2/ABC/SITAAMSIntegrationService",

            "resturl": "http://localhost/api/v1/ABC",

            "windowminimum": -3,

            "windowmaximum": 20,

            "listenerqueue": ".\\private$\\fromamsabc",

            "chunksize":4

        },

        {

            "airport":"XYZ",

            "token":"23892080-3514-480b-4332-567889641",

            "url":"http://localhost/SITAAMSIntegrationService/v2/XYZ/SITAAMSIntegrationService",

            "resturl": "http://localhost/api/v1/XYZ",

            "windowminimum": -3,

            "windowmaximum": 20,

            "listenerqueue": ".\\private$\\fromamsxyz",

            "chunksize":4

        }

    ]

}

 

 

 

airport

The three letter IATA code of the airport used by AMS

Token

An AMS API token that can read flight information

url

The URL of the AMS Native Webservice

Resturl

The URL of the AMS RestAPI service

Windowminimum

The number of days relative to now for the start of the window to process flights (usually negative)

Windowmaximum

The number of days relative to now for the end of the window to process flights

Listenqueue

The AMS Notfication queue for the airport

chunksize

When retrieving flights, only retrieve in chunks of X days

This service was written in the Go programming language as a learning exercise. Even with a large number of flights
and a large number of custom flights per flight, the response performance of the API surprised me. For ever flight returned
the service has to go through every custom field and determine if the user is eligible to receive it and prune out the custom
field if required.
Each flight request does go through the entire list of flights to see if they meet the selection criteria. The flights were
originally stored in a Go Map then a Go Slice, but both proved to be leaky in respect to memory usage. The final
implementation use a custom double linked listed using previous and next pointers which dont leak memory.
There is a list of flight allocations kept for each allocatable resource. Every time a flight is updated, every allocation
is removed for the flight and the allocations recreated. Again I thought this would be inefficient and time consuming, but
testing shows that it is pretty good. I suspect that rather than keeping the allocations for the resources, I could just dynamically
determine the allocations when a request came in.
